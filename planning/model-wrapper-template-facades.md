# Model Wrapper Template Facades

## Purpose

Phase 29 explores an initial wrapper/template layer for Python models generated by Modelwright.

The problem is usability, not calculation correctness. A raw generated model that exposes
`calculate(inputs=None) -> dict[str, value]` over `Sheet!A1` cell references can be validated and
version controlled, but it is not a useful analyst interface by itself. Spreadsheet models carry much
of their meaning through workbook organization: sheets, tables, row labels, column labels, named
ranges, cached values, formulas, formatting conventions, and reporting layouts.

The wrapper layer should help users build custom facades that restore enough of that structure around
generated Python models to support inspection, mutation, execution, and reporting.

## Working Thesis

Generated source is a calculation engine. A wrapper facade is the analyst-facing model boundary.

The facade should make this kind of workflow natural:

- inspect workbook-like structure by sheet, table, row, column, and cell;
- read formula outputs and cached/reference values through named concepts rather than raw cell refs;
- modify model inputs without editing generated source;
- execute the generated model through its existing `calculate(inputs=None)` API;
- report selected outputs as structured tables or named result groups;
- preserve provenance back to workbook cell references.

## Initial Concepts

Candidate public concepts:

- `WrappedModel`: top-level facade around one generated model module or callable.
- `SheetView`: named sheet facade with cells, tables, and reporting regions.
- `TableView`: rectangular range with row labels, column labels, input/output roles, and provenance.
- `CellView`: value/formula/provenance object for one workbook cell.
- `Scenario`: immutable or copy-on-write set of input overrides.
- `Report`: named output selection that can render dictionaries, rows, or simple table payloads.

The initial implementation should stay lightweight and stdlib-oriented. It should not introduce
pandas, NumPy, Arrow, SQLite, or a GUI dependency merely to express the facade contract.

## Metadata Sources

Some structure can come from Modelwright extraction artifacts:

- workbook and sheet names;
- table ranges and table names;
- defined names;
- formula cells and cached values;
- cell references and dependency provenance.

Some structure likely needs user-authored declarations:

- which generated cells are meaningful inputs;
- which sheets/tables are part of a usable public interface;
- row and column labels when workbook layout alone is ambiguous;
- domain names for outputs and reports;
- units, descriptions, scenario defaults, and validation expectations.

The phase should not require perfect automatic workbook semantic recovery. The practical goal is a
template system that lets developers declare the missing structure cleanly.

## Non-Goals

Phase 29 should not build:

- a spreadsheet UI;
- Excel file editing;
- visual formatting reproduction;
- full workbook reconstruction;
- a stable long-term public API guarantee;
- a replacement for compact runtime IR;
- domain-specific FABLE wrapper logic baked into Modelwright core.

FABLE can be used as evidence and pressure-testing material, but wrapper primitives must remain
generic.

## API Direction

The first API should prefer explicit declarations over inference magic. A user should be able to
write a small custom wrapper module around a generated model using Modelwright primitives.

Example shape, not final API:

```python
from modelwright.wrappers import ModelFacade, table

class FoodModel(ModelFacade):
    crop_outputs = table(
        sheet="3_calc_crops",
        range_ref="AO29:AV80",
        rows="A29:A80",
        columns="AO28:AV28",
    )
```

Useful behavior:

- `model.inputs()` returns declared mutable inputs.
- `model.inspect("3_calc_crops!AO29")` returns value/formula/provenance metadata where available.
- `model.set("Inputs!B7", 42)` returns a new scenario or updates an explicit scenario object.
- `model.calculate()` delegates to generated source.
- `model.report("crop_outputs")` returns a structured table payload.

## Risks

- Too much automatic semantic recovery could recreate spreadsheet ambiguity in Python.
- Too much abstraction could hide provenance and make validation harder.
- If wrapper declarations drift from generated model outputs, reports can become misleading.
- A facade API can become sticky quickly; this must remain explicitly alpha until real users exercise
  it.

## Phase 29 Success Criteria

- A tracked planning contract defines the wrapper boundary and non-goals.
- A small initial module provides generic wrapper/template primitives.
- Tests prove the primitives can wrap a generated synthetic model without changing calculation
  behavior.
- Documentation shows how to write a custom wrapper and report structured outputs.
- The release boundary for `0.1.0a4` is explicit: initial wrapper templates, not a full spreadsheet UI
  or stable API.
